# ReSet 문서 동기화 스킬 (reset-doc-sync)

ReSet 프로젝트의 소스 코드 변화를 3개의 핵심 문서에 정확하게 반영하는 반복 가능한 워크플로우를 제공한다.
문서마다 담당 정보의 성격이 다르므로, 각 문서의 고유한 역할과 범위를 이해한 뒤 필요한 것만 갱신한다.

---

## 📋 각 문서의 역할 및 갱신 기준

| 문서 | 담당 정보 | 갱신 트리거 예시 |
|---|---|---|
| `README.md` | 사용자/외부 개발자 대상 기능 소개, 특징, 실행 방법 | 새 기능 추가, 사용 방법 변경 |
| `docs/architecture.md` | 내부 모듈 구조, 클래스 목록, 데이터 흐름, 핵심 메커니즘 | 새 클래스/서비스 추가, 리팩토링 |
| `AGENTS.md` | AI 에이전트용 개발 규칙, 아키텍처 준수 사항, 참조 파일 링크 | 새 규칙 추가, 새 클래스 등장으로 참조 링크 필요 |

---

## 🔄 워크플로우 (4단계)

### 1단계: 소스 코드 스캔

소스 코드의 현재 상태를 파악한다. 다음 항목을 중심으로 읽는다:

- `src/` 하위 모든 프로젝트의 디렉토리 구조
- 새로 추가되었거나 변경된 클래스/인터페이스/서비스 파일
- `appsettings.json` 구성 변경
- `AGENTS.md`, `README.md`, `docs/architecture.md` 현재 내용

**스캔 순서:**
1. `list_dir`으로 `src/` 전체 트리를 파악
2. 최근에 변경된 것으로 보이는 파일이나 사용자가 언급한 파일을 `view_file`로 읽기
3. 3개의 문서를 모두 읽어 현재 상태 파악

### 2단계: 일관성 점검

소스 코드와 문서 간의 괴리를 구체적으로 파악한다:

- `architecture.md`에 없는 새 클래스가 `src/`에 존재하는가?
- `README.md`의 기능 설명이 실제 코드와 다른가?
- `AGENTS.md`의 참조 링크가 존재하지 않는 파일을 가리키는가?

점검 결과를 항목별로 정리하여 **어떤 문서에 어떤 변경이 필요한지** 목록을 만든다.

### 3단계: 업데이트 안(案) 작성

변경이 필요한 문서에 대해서만 업데이트 초안을 작성한다. 변경이 없는 문서는 건너뛴다.

각 문서 업데이트 시 주의사항:

**README.md 작성 원칙:**
- 외부 독자(처음 보는 개발자) 시점에서 작성
- 기능 섹션(`주요 특징`)에 새 기능은 추가, 삭제된 기능은 제거
- 기술 스택, 실행 커맨드가 변경되면 `요구사항` 섹션도 갱신
- 내부 구현 세부사항(클래스명 등)은 포함하지 않음

**docs/architecture.md 작성 원칙:**
- 컴포넌트 테이블에 새 클래스/인터페이스 행 추가 (형식 유지)
- 파일 링크 형식 준수: `[ClassName](file:///home/moondae/git-root/ReSet/src/...)`
- 핵심 메커니즘 섹션은 알고리즘/패턴 변경이 있을 때만 수정
- 기존 설명 스타일(한국어, 기술적 서술)을 유지

**AGENTS.md 작성 원칙:**
- 새 클래스가 생기면 `📂 프로젝트 구조 및 주요 파일 바로가기` 섹션에 링크 추가
- 새 개발 규칙은 기존 범주(보안, 안정성, UI, 파이프라인 등)에 맞는 곳에 추가
- 기존 규칙의 의미를 훼손하지 않도록 추가/수정에만 집중
- 체크리스트(`✅ 에이전트 작업 완료 체크리스트`)에 새 검증 항목이 필요하면 추가

### 4단계: 사용자 검토 후 적용

업데이트 초안을 사용자에게 먼저 보여준다. 다음 형식으로 제안한다:

```
## 📝 업데이트 필요 항목

변경 없음: [문서명들]

변경 필요:
- docs/architecture.md: [이유 요약]
  → [변경 내용 미리보기]
- README.md: [이유 요약]
  → [변경 내용 미리보기]

승인하시면 파일에 바로 적용하겠습니다.
```

사용자가 승인하면 `multi_replace_file_content` 또는 `replace_file_content` 도구로 파일에 적용한다.
사용자가 일부만 승인하거나 수정을 요청하면 해당 부분만 재작성 후 다시 제안한다.

## ⚠️ 주의사항

- **과잉 수정 금지**: 바뀌지 않은 내용은 절대 건드리지 않는다. 문서 전체를 다시 쓰지 말고, 변경이 필요한 섹션만 교체한다.
- **스타일 보존**: 각 문서의 기존 어조(한국어, 이모지 사용 패턴, 목록 형식)를 그대로 유지한다.
- **링크 정확성**: `architecture.md`와 `AGENTS.md`의 파일 링크는 실제 파일이 존재하는 경로로만 작성한다. 존재하지 않는 파일 링크는 추가하지 않는다.
- **단위 테스트 수**: `AGENTS.md`의 체크리스트에 "66개의 단위 테스트" 같은 숫자가 있다면, 실제 테스트 수를 확인(`dotnet test`로 실행하거나 `tests/` 디렉토리 스캔)한 뒤 정확한 숫자로 업데이트한다.
- **사용자 승인 전 적용 금지**: 반드시 초안을 먼저 보여주고 승인 후에 파일을 수정한다.
